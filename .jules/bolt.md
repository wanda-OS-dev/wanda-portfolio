
## 2026-03-25 - [Hoisting Static Allocations in Three.js/React Components]
**Learning:** In Three.js scenes or general React components, calculating static arrays (like a `Float32Array` for random particle positions) using `useMemo` inside the component causes an initial unneeded overhead on mount, and those variables can be unnecessarily garbage collected when the component unmounts. `useMemo` is designed to avoid recalculation *during re-renders*, not necessarily to act as a module-level constant that avoids memory allocation per component instance.
**Action:** When using React with completely static or hardcoded data (like fixed arrays for Three.js coordinates or particle positions), calculate these values outside the component as module-level constants instead of using `useMemo`. This performs the allocation exactly once at module load time rather than on component mount, eliminating unneeded computations and garbage collection overhead during unmounting.

## 2026-03-22 - [Optimizing Render Loop Arrays in Next.js Server & Client Components]
**Learning:** In Next.js pages and components that iterate over static data sets (like `projects.ts`), executing array operations like `Array.prototype.find`, `Array.prototype.findIndex`, or `Array.prototype.slice` directly within the render loop or inside metadata generators creates O(N) operations and unnecessary object allocations on every render pass. This can increase Garbage Collection (GC) pressure and lead to micro-stutters during frequent re-renders or hydration.
**Action:** Instead of inline array operations or using `useMemo` inside the component (which still executes on mount), pre-calculate derivative lists or lookup hash maps (`new Map()`) entirely outside the component or page scope. This converts O(N) searches to O(1) lookups and completely avoids unnecessary heap allocations per render, making both SSR and client hydration more efficient.
